"RAG 구현했다"와 "RAG를 프로덕션에서 운영한다"는 전혀 다른 이야기입니다.

KB국민은행에서 ML 기반 챗봇을 생성형 AI·RAG로 전환하는 작업을 직접 하면서 배운 것들을 정리합니다.

## 1. 청크 크기가 성능을 결정한다

RAG에서 검색 품질은 청크 크기에 크게 좌우됩니다.

너무 크면 → 관련 없는 내용이 섞여서 검색 정확도 저하

너무 작으면 → 컨텍스트가 부족해서 답변이 불완전

금융 FAQ 도메인에서는 질문-답변 단위로 청크를 구성하는 게 단순 문자 수 기반 분할보다 훨씬 효과적이었습니다.

```python
# 단순 분할보다 Q&A 단위 청킹이 효과적
def chunk_by_qa(document: str) -> list[str]:
    """Q: / A: 패턴으로 분할"""
    chunks = []
    qa_pairs = re.split(r'\n(?=Q:)', document)
    for pair in qa_pairs:
        if pair.strip():
            chunks.append(pair.strip())
    return chunks
```

엣지 케이스가 생길 때마다 규칙이 추가됐습니다. 변경 이유와 날짜를 코드 주석으로 남겨두지 않으면 나중에 왜 이 규칙이 있는지 알 수가 없습니다.

## 4. 스트리밍은 선택이 아니다

LLM 응답은 느립니다. 스트리밍 없이 완성된 답변을 기다리게 하면 사용자 이탈이 생깁니다.

타이핑되는 것처럼 보이면 사람들이 기다리는 걸 훨씬 덜 불편해합니다. 실제 응답 시간은 같아도 체감이 완전히 다릅니다.

```typescript
// SSE 스트리밍
const stream = await groq.chat.completions.create({
  model: "llama-3.3-70b-versatile",
  messages: [...],
  stream: true,
});

for await (const chunk of stream) {
  const text = chunk.choices[0]?.delta?.content || "";
  controller.enqueue(encoder.encode(`data: ${JSON.stringify({ text })}\n\n`));
}
```

## 마무리

LLM은 강력하지만 그만큼 세심한 관리가 필요한 것 같습니다. 실험 환경에서 잘 됐던 게 프로덕션에서 예상치 못한 방식으로 실패하는 경우가 있기 떄문입니다.
그 간극을 메우는 것이 결국 AI Engineering 인 것 같습니다.
