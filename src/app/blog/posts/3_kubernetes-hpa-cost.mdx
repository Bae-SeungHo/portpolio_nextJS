AI 서비스를 운영하면 반드시 마주치는 질문이 있습니다.

"이 서버, 새벽에도 다 켜놔야 해?"

KB국민은행 AI 금융비서 서비스에서 이 문제를 실제로 해결했습니다.

## 상황

영업 시간(평일 오전 9시 ~ 오후 6시)에 트래픽이 집중되고, 야간과 주말에는 거의 사용이 없습니다. 그런데 Kubernetes 파드를 24시간 풀 스펙으로 유지하면 비사용 시간대에도 비용이 그대로 나갑니다.

특히 영상 합성 모듈은 CPU/GPU를 많이 잡아먹는 컨테이너라 더 비효율적이었습니다.

## Lambda Scheduling으로 비용 줄이기

AWS Lambda로 시간대별 파드 수를 자동 조절했습니다.

```python
# 평일 오전 8시 50분 → 스케일업 (영업 10분 전 준비)
# 평일 오후 6시 10분 → 스케일다운 (영업 종료 후 10분)
# 주말 → 최소 유지

import boto3

def scale_eks_deployment(replicas: int, deployment: str):
    """EKS Deployment의 replica 수 조정"""
    # kubectl scale 명령어와 동일한 동작
    k8s_client.patch_namespaced_deployment_scale(
        name=deployment,
        namespace="production",
        body={"spec": {"replicas": replicas}}
    )
```

영업 시작 10분 전에 미리 올리는 게 포인트입니다. 파드가 올라오는 데 시간이 걸리기 때문에, 딱 9시에 스케일업하면 고객들이 처음 접속할 때 느려집니다.

## HPA로 트래픽 급증 대응

예외 상황 — 이벤트나 캠페인으로 갑자기 트래픽이 몰릴 때를 대비해서 HPA도 같이 설정했습니다.

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: ai-assistant-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: ai-assistant
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

CPU 사용률 70% 넘으면 파드가 자동으로 늘어나고, 줄어들면 다시 줄어드는 구조입니다.

## 느낀 점

설정 파일 몇 줄이지만, 판단이 필요한 부분들이 있습니다. 10분 전에 스케일업하는 게 맞는지, 70%를 기준으로 하는 게 적절한지 — 이런 건 실제 서비스 트래픽 패턴을 보고 정해야 합니다. 이론이 아니라 운영 경험에서 나오는 감각입니다.
