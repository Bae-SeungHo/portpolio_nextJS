## 개요

KB국민은행에서 기존 ML 기반 챗봇을 생성형 AI로 전환하는 업무를 맡아 진행했습니다.

기존에는 KB국민은행 내부에서 개발한 Bert 모델을 기반으로 Rule-based 답변이 제공되었고,
학습 데이터도 직접 관리를 했었는데요.

이번에 LLM based 생성형 답변 챗봇으로 변경하면서 느낀 점은 다음과 같습니다.

## 기존 ML 챗봇도 나쁜 건 아니다

ML 챗봇의 답변은 명확하죠. 사전에 구축한 답변으로만 제공이 가능하기 때문에 개인화된 질문에 대한 답변이 어렵습니다.
학습 데이터에 없는 표현이 들어오면 무조건 Fallback 이 발생하며,
사전에 구축되어 있지 않다면 "적금 해지하려고요"와 "적금 깨고 싶은데요"를 다른 의도로 분류하는 일이 생깁니다.

하지만 고객 신뢰가 가장 중요한 금융권 서비스에서 잘못된 답변이 나가는 것 또한 매우 Critical 한 문제이기 때문에
기존 ML 챗봇의 예측가능한 답변을 생성하는 것이 가장 중요한 포인트였습니다. 

```text
사용자 입력 → Intent 분류 모델 → 의도 매핑 → 답변 템플릿 반환
```

## LLM으로 전환하며

- 할루시네이션

금융 도메인에서 틀린 정보를 자신 있게 말하면 고객 민원이 되기 때문에
검색 유사도가 낮으면 답변 자체를 막는 임계값 처리가 필요했습니다.
Confidence 가 높은 답변이 아니면 LLM 답변을 사용하지 않고 Fallback 처리했습니다.

- 프롬프트 관리

정합한 답변 제공을 위해 엄격한 프롬프트 관리가 필수적입니다.
모르는 정보에 대해 유추하지 않고 지식베이스에 Retrieve 된 정보만을 가지고 답변할 수 있도록
강력한 시스템 프롬프트를 사용하였습니다.

- 응답 속도

ML 모델 추론은 수십 ms인데 LLM은 수 초입니다.
그래서 답변을 모두 생성하고 반환하지 않고,
스트리밍으로 타이핑되는 것처럼 출력하게 하였습니다.
실제 응답 시간은 같아도 사용자 체감 속도는 많이 차이가 납니다.

## 바꿀 가치가 있었나요?

높은 클라우드 비용과 토큰 이용료가 발생하지만,
ML 챗봇이 "이해하지 못했습니다"로 실패하던 케이스들을 LLM이 자연스럽게 이해하고 지식베이스를 기반으로 답변을 생성하였습니다.

다만, LLM은 질의에 매핑되는 Categorical한 의도가 없기 때문에, 답변에 대한 정합성 검증이 무엇보다 중요하였습니다.

ML과 LLM 두 방식을 상황에 사용하는 하이브리드 구조가 현재로선 가장 좋은 방법이 될 것 같습니다.
