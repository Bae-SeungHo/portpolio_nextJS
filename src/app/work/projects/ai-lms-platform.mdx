금융 AI센터에서 고객컨택혁신부로 부서가 옮겨지면서, 새로운 부서의 업무파악을 하던 도중,
상당히 많은 영역의 단위 업무들이 파편화되고 수기로 관리되고 있다는 점을 발견하게 되었습니다. 

저희 부서에서는 고객과 직원에게 제공하는 여러 채널의 봇 엔진들이 있는데,
(대표적으로 고객용 챗봇,콜봇,직원용 챗봇)
해당 채널마다 담당자와 솔루션이 별도로 존재하고,
각자 로컬 저장소에 학습 데이터와 괸리 대장을 엑셀로 별도로 관리하고 있었습니다.

그러다 보니, 같은 질문인데 채널마다 답이 다르게 나오기도 하고,
신규 상품이 출시되면 모든 채널 담당자가 각자 업데이트하는 불편한 업무 구조를 가지고 있었습니다.

## 개발 과정

처음엔 간단한 단일 페이지의 Flask 대시보드로 시작했는데,
현업 담당자들이 쓰기 시작하면서 요구사항이 쌓였습니다.
"시간대별 질의 통계 보고 싶어요", "비슷한 질문끼리 묶어서 보여주세요" - 이런 식으로요.

그래서 문장 유사도 분석을 위한 임베딩 모델을 직접 PEFT 하여 별도의 추론 서버를 구축하고,
프론트는 템플릿 기반으로 가볍고 빠르게 다중 페이지를 구축 가능한
**FastAPI + HTMX + JavaScript MPA** 구조로 전면 재설계 하였습니다.

## 시스템 구조

```text
운영자 브라우저 (HTMX + JS)
    ↓
내부망 Region Proxy
    ↓
AWS EKS (Kubernetes Deployment)
    ↓
FastAPI 백엔드
    ├── MySQL (학습 데이터 / 대화 로그)
    ├── Redis (메모리 캐싱, 작업 임시 저장)
    └── 임베딩 모델 (질의 유사도 분석)
```

은행망 특성상 타 AWS 계정의 Raw 데이터와 본 프로젝트의 데이터간 통신 설정이 꽤 까다로웠습니다.
방화벽 해제, SG , LB 구성을 인프라팀과 맞추는 데 생각보다 시간이 걸렸던 것 같아요.

## Lesson & Learned
현업의 요구사항을 듣고 필요한 분석 기능을 직접 개발하고, 웹서버 개발과 클라우드 서비스 배포까지 모두 경험해볼 수 있었던 것 같습니다.

특히, 이용량이 많게는 일일 백만건 넘게 유입되는 고객 데이터를 핸들링하고, 스키마가 다른 여러개의 (정말 많은) 채널별 테이블들을 하나의 통합된 기준으로 표준화했던 작업은 힘들었지만 값진 경험이였습니다.
