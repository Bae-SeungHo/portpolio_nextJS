## 왜 만들었나

KB국민은행에서는 고객용 챗봇, 직원용 챗봇, 콜봇 등 여러 채널의 봇 엔진이
각각의 담당자가 로컬 엑셀 파일로 수기 관리하고 있었습니다.

이 구조에는 세 가지 문제가 있었습니다.

첫째, 채널 간 Silo가 발생합니다. 같은 질문이 채널마다 다른 답변을 내는 일이 생겼습니다.
둘째, 변경 이력 추적이 어렵습니다. 언제, 누가, 무엇을 바꿨는지 파악하기 어려웠습니다.
셋째, 신규 상품 출시 등 변경 사항을 모든 채널에 빠르게 반영하기 어려웠습니다.

이 문제를 직접 해결하기 위해 통합 AI 학습관리 플랫폼을 설계하고 개발했습니다.

## 어떻게 만들었나

초기에는 단일 Flask 기반 대시보드로 시작했습니다.
현업 담당자의 피드백을 수렴하며 요구사항이 구체화됐고,
빈출 질의 유형 분석, 시간대별·연령별 질의 패턴 분석 등
실무에서 필요한 기능들이 추가됐습니다.

이후 FastAPI 기반 백엔드와 HTMX + JavaScript MPA 구조로 전환하고,
질의 분석을 위한 임베딩 모델도 추가로 학습하여 통합했습니다.

최종 시스템은 두 가지 레이어로 구성됩니다.

메인 페이지에서는 전체 채널을 통합 모니터링합니다.
서브 페이지에서는 질의 분석, 학습 관리대장 자동화, 형상 관리를 지원합니다.

배포는 AWS EKS 환경에서 Kubernetes Deployment 단위로 구성했습니다.
은행망 내부·외부와 통신하기 위한 Load Balancer, NAT Gateway, Security Group도
인프라 팀과 협업해 설계했습니다.

## 기술 스택
```Backend  : FastAPI (Python)
Frontend : HTMX + JavaScript (MPA)
DB       : MySQL
Infra    : AWS EKS, Kubernetes, Load Balancer, NAT Gateway
AI       : 임베딩 모델 (질의 클러스터링·유사도 분석)
```

## 아키텍처

```
운영자 → Web UI (HTMX + JS)
↓
FastAPI B/E
↓
┌─────────────────┐
│ MySQL (학습 DB)  │
│ 임베딩 모델      │
│ 분석 엔진       │
└─────────────────┘
↓
AWS EKS (Kubernetes)
LB / NAT / SG
```

## 결과

운영관리 인력을 절반으로 감축했습니다.
신규 상품 출시 시 변경 사항을 신속하게 전 채널에 반영할 수 있는 체계가 마련됐습니다.
채널 간 답변 정합성이 높아지고, 변경 이력 추적이 가능해졌습니다.
현업 담당자들로부터 긍정적인 평가를 받았습니다.

## 느낀 점

A-Z로 혼자 설계하고 개발한 프로젝트였습니다.
대규모 고객 데이터를 ETL하고 통합 모니터링하는 플랫폼을 처음부터 끝까지 구축하면서,
아키텍처 결정의 중요성과 현업 사용자 관점으로 개발하는 것의 가치를 다시 배웠습니다.
